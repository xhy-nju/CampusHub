# CampusHub 开发流程清单

本文档用于指导单人开发 CampusHub。开发不按固定周数推进，而按“可运行闭环”推进：每个阶段都产出一个能验证的版本。

## 0. 总体原则

- 先打通主流程，再补完整功能。
- 先做后端核心业务，再做前端页面联调。
- 先做 P0，P0 稳定后再做 P1，最后做 P2。
- 每完成一个阶段都要更新 README、接口文档和测试数据。
- 不为了“看起来完整”提前做复杂推荐、即时通讯或 CD。

## 1. 功能优先级

### P0 必做

- 用户注册、登录、个人资料
- 需求发布、列表、筛选、详情
- 接单、确认、进行中、完成的订单流程

### P1 应做

- 双方评价
- 信用分与信用等级
- 站内通知
- 管理员基础后台

### P2 可选

- 智能匹配与推荐
- 一对一即时通讯
- 数据统计看板增强

## 2. 推荐开发顺序

### 阶段 1：项目地基

- [x] 初始化 Git 仓库、`.gitignore`、CI、Docker Compose
- [x] 确定技术栈：Vue、Element Plus、Pinia、Axios、Spring Boot、MyBatis、JWT、PostgreSQL、Redis
- [x] 建立项目路线文档
- [x] 建立初版数据库 schema
- [x] 初始化 Spring Boot 后端工程
- [ ] 初始化 Vue 前端工程

完成标准：

- `docker compose up -d` 可以启动 PostgreSQL 和 Redis
- `database/init/001_schema.sql` 可以初始化核心表
- README 能说明项目结构和下一步开发入口

### 阶段 2：后端基础设施

- [x] 创建 Spring Boot 3.x 项目
- [x] 配置 Java 17、Maven、PostgreSQL Driver、MyBatis、Spring Security、Redis
- [x] 配置 `application.yml` 和 `local` profile
- [ ] 建立后端包结构：
  - `common`
  - `config`
  - `security`
  - `controller`
  - `service`
  - `mapper`
  - `entity`
  - `dto`
  - `vo`
  - `exception`
- [x] 实现统一响应结构 `ApiResponse<T>`
- [ ] 实现全局异常处理
- [ ] 实现参数校验
- [ ] 实现分页响应
- [ ] 实现 JWT 签发、解析和鉴权过滤器

完成标准：

- 后端可以启动
- `/api/health` 返回正常
- 未登录访问受保护接口会返回 401

### 阶段 3：用户模块

- [ ] 用户注册
- [ ] 用户登录
- [ ] 获取当前用户信息
- [ ] 修改个人资料
- [ ] 密码 BCrypt 加密
- [ ] 用户状态校验：正常、禁用
- [ ] 管理员角色预留

完成标准：

- 可以注册并登录
- 登录后能拿到当前用户信息
- Token 过期或缺失时接口能正确拒绝

### 阶段 4：需求模块

- [ ] 分类查询
- [ ] 发布需求
- [ ] 需求列表
- [ ] 需求详情
- [ ] 按分类筛选
- [ ] 按状态筛选
- [ ] 按关键词搜索
- [ ] 修改自己的需求
- [ ] 取消自己的需求
- [ ] 禁止接自己的需求

完成标准：

- 一个用户可以发布需求
- 另一个用户可以浏览需求详情
- 需求状态清晰可控

### 阶段 5：订单流程

- [ ] 服务方申请接单
- [ ] 需求方确认服务方
- [ ] 订单进入进行中
- [ ] 服务方标记完成
- [ ] 需求方确认完成
- [ ] 订单进入待评价
- [ ] 查询我发布的订单
- [ ] 查询我接取的订单
- [ ] 状态流转校验

建议状态流：

```text
open -> pending_confirm -> in_progress -> provider_done -> completed
```

完成标准：

- 从发布需求到订单完成可以完整跑通
- 后端能拒绝非法状态流转
- 订单双方只能操作自己的订单

### 阶段 6：前端基础工程

- [ ] 创建 Vue 3 + Vite 项目
- [ ] 安装 Element Plus、Pinia、Axios、Vue Router
- [ ] 配置 API base URL
- [ ] 配置 Axios 拦截器
- [ ] 配置登录 Token 存储
- [ ] 配置路由守卫
- [ ] 建立页面布局

完成标准：

- 前端可以启动
- 登录状态可以持久化
- API 错误有统一提示

### 阶段 7：前端主流程页面

- [ ] 登录页
- [ ] 注册页
- [ ] 首页 / 需求列表页
- [ ] 需求详情页
- [ ] 发布需求页
- [ ] 我的发布页
- [ ] 我的接单页
- [ ] 订单详情页
- [ ] 个人中心页

完成标准：

- 可以通过页面完成 P0 全流程
- 核心操作路径不超过 3 步

### 阶段 8：评价与信用体系

- [ ] 双方评价接口
- [ ] 防止重复评价
- [ ] 评分 1-5 星
- [ ] 文字评价
- [ ] 信用分变更记录
- [ ] 用户信用分展示
- [ ] 前端评价页面

初版信用规则：

- 完成订单：双方各 +2
- 获得 5 星评价：被评价方 +3
- 取消已确认订单：责任方 -5
- 管理员处罚：-10

完成标准：

- 已完成订单双方可以互评
- 信用分变化有记录可查

### 阶段 9：站内通知

- [ ] 接单通知
- [ ] 确认通知
- [ ] 完成通知
- [ ] 评价通知
- [ ] 未读数量
- [ ] 已读标记
- [ ] 通知中心页面

完成标准：

- 关键业务状态变化时能生成站内通知
- 用户能查看未读通知

### 阶段 10：管理后台

- [ ] 管理员登录识别
- [ ] 用户列表
- [ ] 禁用 / 启用用户
- [ ] 需求审核
- [ ] 订单统计
- [ ] 分类统计

完成标准：

- 管理员可以处理用户和需求
- 后台有基础统计数据

### 阶段 11：推荐与即时通讯

推荐模块：

- [ ] 基于分类的推荐
- [ ] 基于历史接单行为的推荐
- [ ] 基于地点和时间的排序

即时通讯：

- [ ] 订单内一对一消息
- [ ] 消息列表
- [ ] 未读消息数
- [ ] 初版可用轮询，后续再考虑 WebSocket

完成标准：

- P2 功能不影响 P0/P1 主流程稳定性

### 阶段 12：测试、安全和部署准备

- [ ] 后端核心 service 单元测试
- [ ] 认证和权限测试
- [ ] 订单状态流转测试
- [ ] 前端主流程手动测试
- [ ] SQL 注入和越权访问检查
- [ ] XSS 风险检查
- [ ] README 补充本地启动说明
- [ ] 截图保存到 `screenshots/`
- [ ] 项目稳定后再配置 CD

完成标准：

- CI 通过
- P0/P1 主流程可演示
- 项目文档能指导他人本地运行

## 3. 当前立即执行任务

下一步建议按以下顺序做：

1. 验证 Spring Boot 后端工程、PostgreSQL、Redis 能一起启动
2. 实现用户注册 / 登录
3. 实现需求分类和需求发布
4. 初始化 Vue 前端工程
5. 做登录页和需求列表页

这套顺序的好处是：每一步都能运行验证，不会长期停留在“只有代码但没有闭环”的状态。
